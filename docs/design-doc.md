# Design: MCT2 — Measure Calculation Tool v2

Generated by /office-hours on 2026-03-22 Branch: unknown Repo: bellese/mct2 Status: APPROVED Mode: Startup (Intrapreneurship)

## Problem Statement

Healthcare organizations — particularly ACOs participating in Medicare Shared Savings Program (MSSP) — are required to calculate digital quality measures (dQMs) defined as FHIR Measure resources using Clinical Quality Language (CQL). Most organizations lack the technical infrastructure or expertise to perform these calculations in-house, forcing them to pay vendors $300–$4,000 per provider annually. For organizations with 500–4,000 providers, this adds up to $1–3 million per year in recurring costs.

MCT2 provides a free, open-source orchestration utility that sits between an organization's clinical data repository (FHIR server) and a measure calculation engine (HAPI FHIR with $evaluate-measure), enabling quality improvement staff to calculate dQMs without vendor dependency.

## Demand Evidence

- One ACO spent $250,000 with Infor specifically for FHIR-based measure computation  
- Four additional ACOs confirmed $300–$4,000/provider annual costs for MSSP measure calculations  
- Each ACO serves 500–4,000 providers  
- Consensus across five ACOs: $1–3M in annual vendor costs for measure calculation  
- CMS is actively mandating transition to dQMs, increasing urgency across all quality reporting programs

## Status Quo

Organizations currently outsource dQM calculation to vendors (e.g., Infor, Smile Digital Health, Firely) at significant cost. The open-source landscape offers low-level engines (CQL execution engines, CQF Ruler) and one prototype (the original MCT), but nothing production-ready and simple enough for non-technical quality improvement staff to operate independently.

The original MCT (cqframework/mct) is a Java/React prototype that demonstrates the concept but is explicitly "not suitable or intended for production use" — it lacks authentication, has performance limitations (\~1 sec/patient), and allows arbitrary client modifications to reports.

## Target User & Narrowest Wedge

**Target user:** Quality improvement staff at ACOs, hospitals, and physician practices who are responsible for MSSP and other quality reporting programs. These are healthcare professionals, not software engineers.

**Narrowest wedge:** A docker-compose-deployable tool that:

1. Connects to an existing clinical data repository (FHIR server)  
2. Orchestrates measure calculation via HAPI FHIR's $evaluate-measure  
3. Displays results at both aggregate and patient level

This wedge replaces the most expensive part of the vendor relationship — the actual measure computation and result inspection — while leaving data ingestion, EHR integration, and measure submission to existing workflows.

## Constraints

1. **Single docker compose command** — installation and startup must be trivially simple  
2. **MCT2 does NOT calculate measures** — it delegates to an external HAPI FHIR instance (bundled by default) that performs $evaluate-measure  
3. **Two responsibilities only:** orchestration ($gather-like pattern — see Recommended Approach for definition) and result inspection  
4. **Free and open source** — the entire value proposition depends on this  
5. **Non-technical users** — the UI must be usable by quality improvement staff, not developers

## Premises

1. The target user has access to a FHIR-based clinical data repository but lacks tooling/expertise to run measure calculations against it — **CONFIRMED**  
2. HAPI FHIR's $evaluate-measure is mature enough to serve as the calculation engine — **CONFIRMED**  
3. The primary value is removing vendor dependency ($300–$4,000/provider/year) — **CONFIRMED**  
4. Single docker compose deployment is the right delivery model for v1 — **CONFIRMED**  
5. The original MCT's $gather-like orchestration pattern is the right approach — **CONFIRMED**

## Approaches Considered

### Approach A: Python \+ React (Selected)

Python/FastAPI backend, React frontend, PostgreSQL for job tracking and result caching. Backend handles $gather-like orchestration: fetching patient data from the clinical data repository, submitting to HAPI FHIR for $evaluate-measure, and storing MeasureReport results.

- **Effort:** M (human: \~3 weeks / CC: \~2 hours)  
- **Risk:** Low  
- **Pros:** Python accessible to health IT teams, FastAPI lightweight, strong FHIR library ecosystem (fhir.resources), easy to extend  
- **Cons:** Additional PostgreSQL dependency, two languages (Python \+ JS)  
- **Completeness:** 9/10

### Approach B: Node.js \+ React

Express backend, React frontend, SQLite. Single language across the stack, one fewer Docker container.

- **Effort:** M (human: \~3 weeks / CC: \~2 hours)  
- **Risk:** Low  
- **Pros:** Single language, simpler for contributors, fewer containers  
- **Cons:** JS FHIR libraries less mature, Node less common in health IT  
- **Completeness:** 8/10

### Approach C: Go \+ React

Single Go binary with embedded SQLite, React frontend. Smallest Docker image, fastest startup.

- **Effort:** L (human: \~5 weeks / CC: \~3 hours)  
- **Risk:** Medium — fewer FHIR libraries, harder for health IT contributors  
- **Pros:** Tiny container, fast, minimal dependencies  
- **Cons:** Steeper contributor learning curve, weaker FHIR ecosystem  
- **Completeness:** 7/10

## Recommended Approach

**Approach A: Python \+ React** — chosen because Python is the most accessible language for health IT quality improvement teams, has the strongest FHIR library ecosystem, and FastAPI keeps the backend lightweight. PostgreSQL adds one container but provides robust job tracking for large patient populations (4,000+ providers).

### What "$gather-like" Means

The original MCT implements a FHIR `$gather` operation pattern: it collects (gathers) patient clinical data from a source FHIR server, packages it appropriately, and submits it to a measure calculation engine for evaluation. MCT2 follows this same pattern — it acts as the intermediary that pulls data from the clinical data repository, forwards it to HAPI FHIR for `$evaluate-measure`, and collects the resulting MeasureReport resources. The user never interacts with either FHIR server directly.

### Prerequisites

- **Hardware:** 16 GB RAM recommended (two HAPI FHIR JPA instances require \~4 GB each for CQL evaluation and data serving; PostgreSQL, Python backend, and React frontend share the remainder). 8 GB is the absolute minimum but may encounter OOM issues under heavy load. 4 CPU cores recommended. 20 GB disk for persistent data.  
- **Software:** Docker Engine 24+ and Docker Compose v2+ (installed by the organization's IT staff; quality improvement staff operate the web UI, not Docker directly)  
- **Network:** A bundled default CDR is included for immediate use. For production, access to the organization's clinical data repository (FHIR server endpoint) is required

### Architecture

```
docker compose up
  |
  +-- mct2-frontend (React)
  |     Web UI for administration, triggering calculations,
  |     and inspecting MeasureReport results
  |
  +-- mct2-backend (Python/FastAPI)
  |     Orchestrator: $gather-like pattern
  |     - Connects to CDR (default bundled or external)
  |     - Fetches patient/resource data
  |     - Submits to measure engine for $evaluate-measure
  |     - Stores and serves MeasureReport results
  |     REST API for frontend
  |
  +-- mct2-db (PostgreSQL)
  |     Job tracking, configuration, cached MeasureReports
  |
  +-- hapi-fhir-cdr (HAPI FHIR JPA Server — Default CDR)
  |     Bundled clinical data repository
  |     Pre-loaded with test data for out-of-box experience
  |     Users configure MCT2 to point to their own CDR in production
  |
  +-- hapi-fhir-measure (HAPI FHIR JPA Server — Measure Engine)
        Measure calculation engine
        Performs $evaluate-measure
        Hosts Measure resources and CQL libraries
```

**Two HAPI FHIR instances, two distinct roles:**

- **hapi-fhir-cdr:** Acts as the default clinical data repository. Ships with sample patient data so users can run their first measure calculation immediately after `docker compose up`. In production, users configure MCT2 to point to their organization's real CDR instead — the bundled instance becomes unused or can be stopped.  
- **hapi-fhir-measure:** The measure calculation engine. Hosts FHIR Measure resources and CQL libraries, and performs `$evaluate-measure`. This instance is always used, regardless of which CDR is configured.

### Key Workflows

**1\. Configuration**

- Out of the box, MCT2 points to the bundled CDR (hapi-fhir-cdr) — no configuration needed to start exploring  
- User can reconfigure to point to their organization's CDR via the web UI (FHIR server endpoint URL \+ optional auth credentials)  
- User selects Measure resources to evaluate (loaded into hapi-fhir-measure)  
- User configures measurement period and subject parameters

**2\. Measure Loading**

- User obtains FHIR Measure bundles (Measure resource \+ CQL libraries) from the CMS eCQI Resource Center ([https://ecqi.healthit.gov/fhir](https://ecqi.healthit.gov/fhir)) or their quality program's measure repository  
- User uploads these bundles to the measure engine (hapi-fhir-measure) via the MCT2 web UI  
- MCT2 backend POSTs the resources as a FHIR Bundle to hapi-fhir-measure's REST API  
- Loaded measures persist across restarts (hapi-fhir-measure data is volume-mounted)  
- UI displays currently loaded measures with status indicators

**3\. Orchestration ($gather-like)**

- MCT2 backend queries clinical data repository for patient resources matching the measure's data requirements  
- Invokes HAPI FHIR's `$evaluate-measure` once per patient per measure. For large populations, patients are processed in configurable batches (default: 100 patients per batch) with parallel workers (default: 4 concurrent `$evaluate-measure` calls). Back-of-envelope: at \~1 sec/patient with 4 workers, 4,000 patients takes \~17 minutes; with connection pooling and HAPI JPA warm cache, sub-15 minutes is achievable  
- **Note on providers vs. patients:** The demand evidence cites costs per *provider* (physician). Each provider has a panel of patients; the actual patient volume per measure run depends on the measure and panel sizes. The 4,000-patient target is a reasonable starting point for a single measure run; larger populations may require longer runs  
- Tracks job progress (queued, running, complete, failed) with per-batch granularity  
- Stores resulting MeasureReport resources in PostgreSQL

**3a. Error Handling & Retries**

- Individual batch failures do not abort the entire job — failed batches are retried up to 3 times with exponential backoff  
- Partial results are preserved: if 38 of 40 batches succeed, the user sees results for 3,800 patients and a clear indicator that 2 batches (200 patients) failed  
- Failures surface in the UI with error details (HTTP status, FHIR OperationOutcome messages)  
- All orchestration activity is logged with structured JSON logging; a `/health` endpoint on the backend reports service status

**4\. Result Inspection**

- Aggregate view: population-level summary across all measures  
- Patient-level drill-down: individual MeasureReport details  
- Filter/sort by measure, period, population criteria (numerator, denominator, etc.)

      - POSTGRES_DB=mct2
      - POSTGRES_USER=mct2
      - POSTGRES_PASSWORD=mct2

  hapi-fhir-cdr:
    image: hapiproject/hapi:v7.6.0

    image: hapiproject/hapi:v7.6.0

volumes:
  pgdata:
  cdrdata:
  measuredata:

## Open Questions

1. **Authentication:** V1 should support at minimum static Bearer token and Basic Auth for connecting to clinical data repositories — most production FHIR servers require this. SMART on FHIR/OAuth2 deferred to a later release. Exact auth configuration UX TBD.  
2. **Report submission:** Does MCT2 need to support submitting MeasureReports to a receiving system (e.g., CMS), or is inspection sufficient for v1?  
3. **Historical runs:** Should users be able to run the same measure for multiple measurement periods (e.g., Q1 vs Q2) and compare results over time? This affects data model design.

## Success Criteria

- Quality improvement staffer can install MCT2 with a single `docker compose up` command  
- Can configure a connection to their clinical data repository via the web UI  
- Can trigger measure calculation for selected dQMs and a defined measurement period  
- Can inspect aggregate and patient-level results through the web UI  
- Total setup-to-first-result time under 30 minutes (excluding data loading)  
- Zero vendor dependency for the calculation workflow

## Dependencies

- HAPI FHIR JPA Server x2 (bundled: one as default CDR, one as measure calculation engine)  
- For production use: access to the organization's FHIR-compliant clinical data repository (replaces the bundled CDR)  
- FHIR Measure resources and CQL libraries for the desired dQMs  
- Docker and Docker Compose on the host machine

### Responsive Behavior

- **Desktop (1280px+):** Full layout as shown — sidebar \+ main content  
- **Tablet (768-1279px):** Collapsible sidebar (hamburger toggle), main content fills width  
- **Mobile (\< 768px):** Bottom tab navigation (Measures, Jobs, Results), settings via hamburger menu. Data tables become card lists. Patient drill-down is a full-screen view, not a slide-out panel.  
- **Touch targets:** 44px minimum on all interactive elements  
- **Data tables on small screens:** Horizontal scroll with sticky first column (patient name/measure name)

### Accessibility Requirements

- **Target:** WCAG 2.1 AA (required for healthcare tools under Section 508\)  
- **Color contrast:** 4.5:1 minimum for normal text, 3:1 for large text  
- **Keyboard navigation:** All interactive elements reachable via Tab. Sidebar navigable with arrow keys. Escape closes modals/panels.  
- **Screen readers:** ARIA landmarks for sidebar, main content, and top bar. Live regions for job progress updates and toast notifications.  
- **Focus indicators:** Visible focus ring on all interactive elements (not browser default — custom 2px ring)  
- **No color-only indicators:** Status icons (healthy/unhealthy, success/failure) use both color AND icon shape

## Reviewer Concerns

The following items were flagged during adversarial review but are implementation-level details to resolve during development rather than design-level decisions:

1. **HAPI FHIR data path:** The `hapidata:/data/hapi` volume mount assumes HAPI's default data directory. The actual path depends on HAPI's datasource configuration (embedded H2 vs. external PostgreSQL). Verify during implementation.  
2. **API contract:** No REST endpoint list or database schema is specified in this design doc. These should be defined during implementation planning (e.g., via `/plan-eng-review`).  
3. **Measure versioning:** How multiple versions of the same measure coexist in HAPI FHIR needs investigation during implementation.
